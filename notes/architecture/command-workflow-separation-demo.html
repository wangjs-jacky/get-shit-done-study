GSD Framework Architecture

Command 与 Workflow 分离架构

GSD 框架采用两层架构：Command 负责声明，Workflow 负责实现。让命令既易于理解，又易于维护。

核心架构：两层职责

C
Command Layer
声明层 · "做什么"

薄薄的入口层，定义元数据和依赖。

  • name, description
  • allowed-tools（最小权限原则）
  • execution_context（依赖引用）
W
Workflow Layer
实现层 · "怎么做"

厚厚的实现层，包含具体步骤和逻辑。

  • step 定义
  • conditions 条件
  • bash 命令
  • 交互逻辑

文件结构对比

Command 文件结构

commands/gsd/update.md 
---
name: gsd:update
description: Update GSD to latest version
allowed-tools:
  - Bash
  - AskUserQuestion
---

<objective>
  Check for updates, install if available...
</objective>

<execution_context>
  @~/.claude/get-shit-done/workflows/update.md
</execution_context>

Workflow 文件结构

workflows/update.md 
<purpose>Check for GSD updates...</purpose>

<step name="get_installed_version">
  检测本地/全局安装...
  ```bash
  # 具体的 bash 命令
  ```
</step>

<step name="check_latest_version">
  ```bash
  npm view get-shit-done-cc version
  ```
</step>

设计亮点

01
关注点分离
接口与实现解耦
02
工具权限控制
每个命令明确声明需要的工具
03
可复用
一个 workflow 可被多个 command 引用
04
可维护
修改实现不影响命令接口
