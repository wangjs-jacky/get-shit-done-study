Command 与 Workflow 分离架构

GSD 框架采用两层架构：Command 负责声明，Workflow 负责实现

// 问题

框架如何让命令既易于理解，又易于维护？

// 核心思想

将"做什么"（Command）与"怎么做"（Workflow）分离
> COMMAND
薄薄的入口层，定义元数据和依赖
> WORKFLOW
厚厚的实现层，包含具体步骤和逻辑
层级位置职责内容
Commandcommands/gsd/*.md声明"做什么"name, description, allowed-tools
Workflowworkflows/*.md定义"怎么做"steps, conditions, bash

// Command 结构

YAML
---
name: gsd:update
description: Update GSD to latest version
allowed-tools:
  - Bash
  - AskUserQuestion
---

// 设计亮点

01 / 关注点分离
接口与实现解耦
02 / 工具权限控制
最小权限原则
03 / 可复用
一个 workflow 多个 command
04 / 可维护
修改实现不影响接口
