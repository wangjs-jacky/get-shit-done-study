Command 与 Workflow 分离架构

GSD 框架采用两层架构
Command 负责声明，Workflow 负责实现

问题

框架如何让命令既易于理解，又易于维护？

核心思想

将"做什么"与"怎么做"分离
Command
薄薄的入口层
定义元数据和依赖
Workflow
厚厚的实现层
包含具体步骤和逻辑

两层职责

层级位置职责
Commandcommands/gsd/*.md声明"做什么"
Workflowworkflows/*.md定义"怎么做"

Command 结构

yaml
---
name: gsd:update
description: Update GSD to latest version
allowed-tools:
  - Bash
  - AskUserQuestion
---

npm 包 → install.js → 本地目录
  │           │           │
  │           │           └── commands/
  │           │           └── workflows/
  │           │           └── agents/
  │           │
  │           └── copyWithPathReplacement()
  │
  └── get-shit-done-cc@latest

设计亮点

  • 关注点分离：接口与实现解耦
  • 工具权限控制：最小权限原则
  • 可复用：一个 workflow 多个 command
  • 可维护：修改实现不影响接口

可复用场景

  • jacky-skills：skill 定义与实现分离
  • 声明式接口 + 命令式实现
