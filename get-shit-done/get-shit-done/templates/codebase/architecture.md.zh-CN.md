# 架构模板

用于 `.planning/codebase/ARCHITECTURE.md` - 捕获概念性代码组织。

**目的：** 在概念层面记录代码如何组织。与 STRUCTURE.md（显示物理文件位置）相互补充。

<good_examples>
```markdown
# 架构

**分析日期：** 2025-01-20

## 模式概览

**整体：** 带插件系统的CLI应用程序

**关键特征：**
- 单个可执行文件，包含子命令
- 基于插件的可扩展性
- 基于文件的状态（无数据库）
- 同步执行模型

## 层次结构

**命令层：**
- 目的：解析用户输入并路由到适当的处理器
- 包含：命令定义、参数解析、帮助文本
- 位置：`src/commands/*.ts`
- 依赖：业务逻辑的服务层
- 被使用：CLI入口点（`src/index.ts`）

**服务层：**
- 目的：核心业务逻辑
- 包含：FileService、TemplateService、InstallService
- 位置：`src/services/*.ts`
- 依赖：文件系统工具、外部工具
- 被使用：命令处理器

**工具层：**
- 目的：共享辅助程序和抽象
- 包含：文件I/O包装器、路径解析、字符串格式化
- 位置：`src/utils/*.ts`
- 依赖：仅Node.js内置模块
- 被使用：服务层

## 数据流

**CLI命令执行：**

1. 用户运行：`gsd new-project`
2. Commander解析参数和标志
3. 调用命令处理器（`src/commands/new-project.ts`）
4. 处理器调用服务方法（`src/services/project.ts` → `create()`）
5. 服务读取模板、处理文件、写入输出
6. 结果记录到控制台
7. 进程以状态码退出

**状态管理：**
- 基于文件：所有状态都在`.planning/`目录中
- 无持久内存状态
- 每个命令执行都是独立的

## 核心抽象

**服务：**
- 目的：为领域封装业务逻辑
- 示例：`src/services/file.ts`、`src/services/template.ts`、`src/services/project.ts`
- 模式：类似单例（作为模块导入，不实例化）

**命令：**
- 目的：CLI命令定义
- 示例：`src/commands/new-project.ts`、`src/commands/plan-phase.ts`
- 模式：Commander.js命令注册

**模板：**
- 目的：可重用的文档结构
- 示例：PROJECT.md、PLAN.md模板
- 模式：带有替换变量的Markdown文件

## 入口点

**CLI入口：**
- 位置：`src/index.ts`
- 触发器：用户运行`gsd <command>`
- 职责：注册命令、解析参数、显示帮助

**命令：**
- 位置：`src/commands/*.ts`
- 触发器：来自CLI的匹配命令
- 职责：验证输入、调用服务、格式化输出

## 错误处理

**策略：** 抛出异常，在命令级别捕获，记录并退出

**模式：**
- 服务抛出带有描述性消息的Error
- 命令处理器捕获，将错误记录到stderr，退出(1)
- 验证错误在执行前显示（快速失败）

## 横切关注点

**日志：**
- Console.log用于正常输出
- Console.error用于错误
- Chalk用于彩色输出

**验证：**
- 配置文件解析的Zod模式
- 命令处理器中的手动验证
- 无效输入快速失败

**文件操作：**
- fs-extra上的FileService抽象
- 所有操作前验证路径
- 原子写入（临时文件+重命名）

---

*架构分析：2025-01-20*
*主要模式变化时更新*
```

<guidelines>
**ARCHITECTURE.md中应包含的内容：**
- 整体架构模式（单体、微服务、分层等）
- 概念层次结构及其关系
- 数据流/请求生命周期
- 核心抽象和模式
- 入口点
- 错误处理策略
- 横切关注点（日志、认证、验证）

**这里不应包含的内容：**
- 详尽的文件列表（那是STRUCTURE.md）
- 技术选择（那是STACK.md）
- 行代码级演练（ defer到代码读取）
- 特定功能的实现细节

**文件路径受欢迎：**
将文件路径作为具体抽象示例包含。使用反引号格式：`src/services/user.ts`。这使得架构文档对规划时的Claude具有可操作性。

**填写此模板时：**
- 读取主要入口点（index、server、main）
- 通过读取导入/依赖识别层次结构
- 跟踪典型的请求/命令执行
- 注意重复模式（服务、控制器、仓库）
- 保持描述性概念，而不是机械性

**阶段规划时有用：**
- 添加新功能（它在哪一层中？）
- 重构（理解当前模式）
- 识别在哪里添加代码（哪一层处理X？）
- 理解组件之间的依赖关系
</guidelines>
