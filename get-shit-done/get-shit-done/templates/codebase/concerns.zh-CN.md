# 代码库问题模板

用于 `.planning/codebase/CONCERNS.md` 的模板 - 记录已知问题和需要关注的领域。

**目的:** 提出关于代码库的可操作警告。重点关注"进行更改时需要注意什么"。

<good_examples>
```markdown
# 代码库问题

**分析日期:** 2025-01-20

## 技术债务

**React 组件中的数据库查询:**
- 问题: 15+ 个页面组件中直接使用 Supabase 查询，而不是服务器操作
- 文件: `app/dashboard/page.tsx`、`app/profile/page.tsx`、`app/courses/[id]/page.tsx`、`app/settings/page.tsx`（以及 `app/` 中的另外 11 个）
- 原因: MVP 阶段快速原型设计
- 影响: 无法正确实施 RLS，向客户端暴露数据库结构
- 修复方法: 将所有查询移动到 `app/actions/` 中的服务器操作，添加适当的 RLS 策略

**手动 webhook 签名验证:**
- 问题: 复制的 Stripe webhook 验证代码在 3 个不同的端点中
- 文件: `app/api/webhooks/stripe/route.ts`、`app/api/webhooks/checkout/route.ts`、`app/api/webhooks/subscription/route.ts`
- 原因: 每个 webhook 在没有抽象的情况下临时添加
- 影响: 容易在新 webhooks 中遗漏验证（安全风险）
- 修复方法: 创建共享的 `lib/stripe/validate-webhook.ts` 中间件

## 已知错误

**订阅更新中的竞态条件:**
- 症状: 成功付款后 5-10 秒内用户显示为"免费"层级
- 触发: Stripe 结账重定向后，在 webhook 处理前快速导航
- 文件: `app/checkout/success/page.tsx`（重定向处理器）、`app/api/webhooks/stripe/route.ts`（webhook）
- 变通方法: Stripe webhook 最终更新状态（自愈）
- 根本原因: Webhook 处理比用户导航慢，无乐观 UI 更新
- 修复: 在 `app/checkout/success/page.tsx` 重定向后添加轮询

**登出后的不一致会话状态:**
- 症状: 用户登出后被重定向到 /dashboard 而不是 /login
- 触发: 通过移动导航中的按钮登出（桌面端工作正常）
- 文件: `components/MobileNav.tsx`（约第 45 行，登出处理器）
- 变通方法: 手动导航到 /login 有效
- 根本原因: 移动导航组件没有等待 supabase.auth.signOut()
- 修复: 在 `components/MobileNav.tsx` 的登出处理器中添加 await

## 安全考虑

**仅客户端检查管理员角色:**
- 风险: 管理员页面仅从 Supabase 客户端检查 isAdmin，无服务器验证
- 文件: `app/admin/page.tsx`、`app/admin/users/page.tsx`、`components/AdminGuard.tsx`
- 当前缓解: 无（依赖 UI 隐藏）
- 建议: 在 `middleware.ts` 中为管理员路由添加中间件，服务器端验证角色

**未验证的文件上传:**
- 风险: 用户可以将任何文件类型上传到头像存储桶（无大小/类型验证）
- 文件: `components/AvatarUpload.tsx`（上传处理器）
- 当前缓解: Supabase 存储桶限制为 2MB（在仪表板中配置）
- 建议: 在 `lib/storage/validate.ts` 中添加文件类型验证（仅 image/*）

## 性能瓶颈

**/api/courses 端点:**
- 问题: 获取所有课程及其嵌套课程和作者
- 文件: `app/api/courses/route.ts`
- 测量: 50+ 课程时 p95 响应时间为 1.2s
- 原因: N+1 查询模式（每个课程的单独课程查询）
- 改进路径: 在 `lib/db/courses.ts` 中使用 Prisma include 进行急加载，添加 Redis 缓存

**仪表板初始加载:**
- 问题: 挂载时 5 个串行 API 调用的瀑布流
- 文件: `app/dashboard/page.tsx`
- 测量: 在慢速 3G 上 3.5s 才交互
- 原因: 每个组件独立获取自己的数据
- 改进路径: 转换为具有单个并行获取的 Server Component

## 脆弱区域

**身份验证中间件链:**
- 文件: `middleware.ts`
- 为什么脆弱: 4 个不同的中间件函数按特定顺序运行（身份验证 → 角色 → 订阅 → 日志）
- 常见故障: 中间件顺序更改会破坏一切，难以调试
- 安全修改: 更改顺序前添加测试，在注释中记录依赖关系
- 测试覆盖: 中间件链无集成测试（仅单元测试）

**Stripe webhook 事件处理:**
- 文件: `app/api/webhooks/stripe/route.ts`
- 为什么脆弱: 包含 12 个事件类型的巨大 switch 语句，共享事务逻辑
- 常见故障: 添加新事件类型但未处理，错误时部分数据库更新
- 安全修改: 将每个事件处理程序提取到 `lib/stripe/handlers/*.ts`
- 测试覆盖: 12 个事件类型中仅 3 个有测试

## 扩展限制

**Supabase 免费层级:**
- 当前容量: 500MB 数据库、1GB 文件存储、2GB 带宽/月
- 限制: 预计 ~5000 用户时达到限制
- 限制时的症状: 429 速率限制错误，数据库写入失败
- 扩展路径: 升级到 Pro（25 美元/月）扩展到 8GB 数据库、100GB 存储

**服务器端渲染阻塞:**
- 当前容量: 在减慢前约 50 个并发用户
- 限制: Vercel Hobby 计划（10s 函数超时，100GB-hrs/月）
- 限制时的症状: 课程页面 504 网关超时
- 扩展路径: 升级到 Vercel Pro（20 美元/月），添加边缘缓存

## 有风险的依赖

**react-hot-toast:**
- 风险: 未维护（最后更新 18 个月前），React 19 兼容性未知
- 影响: Toast 通知中断，无优雅降级
- 迁移计划: 切换到 sonner（积极维护，类似 API）

## 缺失的关键功能

**付款失败处理:**
- 问题: 订阅付款失败时无重试机制或用户通知
- 当前变通方法: 用户手动重新输入付款信息（如果他们注意到）
- 阻塞: 无法保留信用卡过期的用户，无催收流程
- 实现复杂度: 中等（Stripe webhooks + 邮件流程 + UI）

**课程进度跟踪:**
- 问题: 缺少已完成课程的持久状态
- 当前变通方法: 用户手动跟踪进度
- 阻塞: 无法显示完成百分比，无法推荐下一课程
- 实现复杂度: 低（添加 completed_lessons 连接表）

## 测试覆盖空缺

**付款流程端到端:**
- 未测试的内容: 完整的 Stripe 结账 → webhook → 订阅激活流程
- 风险: 付款处理可能静默损坏（已发生两次）
- 优先级: 高
- 测试难度: 需要 Stripe 测试装置和 webhook 模拟设置

**错误边界行为:**
- 未测试的内容: 组件抛出错误时应用程序的行为
- 风险: 用户的白屏死机，无错误报告
- 优先级: 中
- 测试难度: 需要在测试环境中故意触发错误

---

*问题审核: 2025-01-20*
*在问题修复或新问题时更新*
```

<guidelines>
**CONCERNS.md 应包含的内容:**
- 具有清晰影响和修复方法的技术债务
- 带有重现步骤的已知错误
- 安全漏洞和缓解建议
- 具有测量的性能瓶颈
- 容易损坏的脆弱代码
- 有数字的扩展限制
- 需要关注的依赖
- 阻塞工作流的缺失功能
- 测试覆盖空缺

**这里不应包含的内容:**
- 无证据的意见（"代码很乱"）
- 无解决方案的抱怨（"身份验证 sucks"）
- 未来功能想法（那是产品规划）
- 正常的 TODO（住在代码注释中）
- 正在工作的架构决策
- 次要代码风格问题

**填写此模板时:**
- **始终包含文件路径** - 没有位置的问题是不可操作的。使用反斜杠：`src/file.ts`
- 使用具体测量（"500ms p95" 而不是 "慢"）
- 为错误包含重现步骤
- 建议修复方法，而不仅仅是问题
- 专注于可操作的项目
- 按风险/影响优先排序
- 在问题解决时更新
- 发现新问题时添加

**语气指导:**
- 专业，而非情绪化（"N+1 查询模式" 而不是 "糟糕的查询"）
- 解决方案导向（"修复：添加索引" 而不是 "需要修复"）
- 风险导向（"可能暴露用户数据" 而不是 "安全性不好"）
- 事实性（"3.5s 加载时间" 而不是 "真的慢"）

**在阶段规划中用途:**
- 决定接下来工作什么
- 评估变更风险
- 理解在哪里需要小心
- 优先级排序改进
- 新 Claude 上下文入职
- 计划重构工作

**如何填充此文档:**
探索代理在代码库映射期间检测这些。欢迎手动添加人类发现的问题。这是活动文档，而非投诉列表。
</guidelines>
