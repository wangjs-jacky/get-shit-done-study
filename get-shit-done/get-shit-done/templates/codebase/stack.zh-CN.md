# 技术栈模板

用于 `.planning/codebase/STACK.md` 的模板 - 记录技术基础。

**目的:** 记录运行此代码库的技术栈。重点关注"运行代码时执行什么"。

---

## 文件模板

```markdown
# 技术栈

**分析日期:** [YYYY-MM-DD]

## 语言

**主要:**
- [语言] [版本] - [使用场景: 例如，"所有应用程序代码"]

**次要:**
- [语言] [版本] - [使用场景: 例如，"构建脚本、工具"]

## 运行环境

**环境:**
- [运行时] [版本] - [例如，"Node.js 20.x"]
- [其他需求（如有）]

**包管理器:**
- [管理器] [版本] - [例如，"npm 10.x"]
- 锁定文件: [例如，"存在 package-lock.json"]

## 框架

**核心:**
- [框架] [版本] - [用途: 例如，"Web 服务器"、"UI 框架"]

**测试:**
- [框架] [版本] - [例如，"Jest 用于单元测试"]
- [框架] [版本] - [例如，"Playwright 用于端到端测试"]

**构建/开发:**
- [工具] [版本] - [例如，"Vite 用于打包"]
- [工具] [版本] - [例如，"TypeScript 编译器"]

## 关键依赖

[仅包含理解技术栈所必需的依赖 - 限制在 5-10 个最重要的]

**关键:**
- [包] [版本] - [重要性原因: 例如，"身份验证"、"数据库访问"]
- [包] [版本] - [重要性原因]

**基础设施:**
- [包] [版本] - [例如，"Express 用于 HTTP 路由"]
- [包] [版本] - [例如，"PostgreSQL 客户端"]

## 配置

**环境:**
- [配置方式: 例如，".env 文件"、"环境变量"]
- [关键配置: 例如，"需要 DATABASE_URL、API_KEY"]

**构建:**
- [构建配置文件: 例如，"vite.config.ts、tsconfig.json"]

## 平台要求

**开发:**
- [操作系统需求或"任何平台"]
- [额外工具: 例如，"Docker 用于本地数据库"]

**生产:**
- [部署目标: 例如，"Vercel"、"AWS Lambda"、"Docker 容器"]
- [版本要求]

---

*技术栈分析: [日期]*
*在主要依赖变更后更新*
```

<good_examples>
```markdown
# 技术栈

**分析日期:** 2025-01-20

## 语言

**主要:**
- TypeScript 5.3 - 所有应用程序代码

**次要:**
- JavaScript - 构建脚本、配置文件

## 运行环境

**环境:**
- Node.js 20.x (LTS)
- 无浏览器运行时（仅 CLI 工具）

**包管理器:**
- npm 10.x
- 锁定文件: 存在 `package-lock.json`

## 框架

**核心:**
- 无（原生 Node.js CLI）

**测试:**
- Vitest 1.0 - 单元测试
- tsx - 无构建步骤的 TypeScript 执行

**构建/开发:**
- TypeScript 5.3 - 编译为 JavaScript
- esbuild - 被 Vitest 用于快速转换

## 关键依赖

**关键:**
- commander 11.x - CLI 参数解析和命令结构
- chalk 5.x - 终端输出样式
- fs-extra 11.x - 扩展文件系统操作

**基础设施:**
- Node.js 内置模块 - fs、path、child_process 用于文件操作

## 配置

**环境:**
- 不需要环境变量
- 仅通过 CLI 标志配置

**构建:**
- `tsconfig.json` - TypeScript 编译器选项
- `vitest.config.ts` - 测试运行器配置

## 平台要求

**开发:**
- macOS/Linux/Windows（任何有 Node.js 的平台）
- 无外部依赖

**生产:**
- 作为 npm 包分发
- 通过 npm install -g 全局安装
- 在用户的 Node.js 安装上运行

---

*技术栈分析: 2025-01-20*
*在主要依赖变更后更新*
```

<guidelines>
**STACK.md 应包含的内容:**
- 语言和版本
- 运行时要求（Node、Bun、Deno、浏览器）
- 包管理器和锁定文件
- 框架选择
- 关键依赖（限制在 5-10 个最重要的）
- 构建工具
- 平台/部署要求

**这里不应包含的内容:**
- 文件结构（那是 STRUCTURE.md）
- 架构模式（那是 ARCHITECTURE.md）
- package.json 中的每个依赖（只有关键依赖）
- 实现细节（ defer to code）

**填写此模板时:**
- 检查 package.json 中的依赖
- 从 .nvmrc 或 package.json engines 中记录运行时版本
- 只包含影响理解的依赖（不是每个工具）
- 仅在版本重要时指定版本（破坏性变更、兼容性）

**在阶段规划中用途:**
- 添加新依赖（检查兼容性）
- 升级框架（了解正在使用的工具）
- 选择实现方法（必须与现有技术栈兼容）
- 理解构建要求
</guidelines>
