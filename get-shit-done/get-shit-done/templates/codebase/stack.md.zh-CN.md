# 技术栈模板

用于 `.planning/codebase/STACK.md` - 捕获技术基础。

**目的：** 记录运行此代码库的技术。专注于"代码运行时执行什么"。

---

## 文件模板

```markdown
# 技术栈

**分析日期：** [YYYY-MM-DD]

## 语言

**主要：**
- [语言] [版本] - [用途：如"所有应用代码"]

**次要：**
- [语言] [版本] - [用途：如"构建脚本、工具"]

## 运行时

**环境：**
- [运行时] [版本] - [如"Node.js 20.x"]
- [任何其他要求]

**包管理器：**
- [管理器] [版本] - [如"npm 10.x"]
- 锁文件：[如"package-lock.json存在"]

## 框架

**核心：**
- [框架] [版本] - [目的：如"web服务器", "UI框架"]

**测试：**
- [框架] [版本] - [如"单元测试的Jest"]
- [框架] [版本] - [如"E2E的Playwright"]

**构建/开发：**
- [工具] [版本] - [如"捆绑的Vite"]
- [工具] [版本] - [如"TypeScript编译器"]

## 关键依赖

[只对理解栈重要的依赖 - 限制为5-10个最重要]

**关键：**
- [包] [版本] - [为什么重要：如"认证", "数据库访问"]
- [包] [版本] - [为什么重要]

**基础设施：**
- [包] [版本] - [如"HTTP路由的Express"]
- [包] [版本] - [如"PostgreSQL客户端"]

## 配置

**环境：**
- [如何配置：如".env文件", "环境变量"]
- [关键配置：如"需要DATABASE_URL, API_KEY"]

**构建：**
- [构建配置文件：如"vite.config.ts, tsconfig.json"]

## 平台要求

**开发：**
- [OS要求或"任何平台"]
- [其他工具：如"本地数据库的Docker"]

**生产：**
- [部署目标：如"Vercel", "AWS Lambda", "Docker容器"]
- [版本要求]

---

*栈分析：[日期]*
*主要依赖更改后更新*
```

<good_examples>
```markdown
# 技术栈

**分析日期：** 2025-01-20

## 语言

**主要：**
- TypeScript 5.3 - 所有应用代码

**次要：**
- JavaScript - 构建脚本、配置文件

## 运行时

**环境：**
- Node.js 20.x (LTS)
- 无浏览器运行时（仅CLI工具）

**包管理器：**
- npm 10.x
- 锁文件：`package-lock.json`存在

## 框架

**核心：**
- 无（原生Node.js CLI）

**测试：**
- Vitest 1.0 - 单元测试
- tsx - 无构建步骤的TypeScript执行

**构建/开发：**
- TypeScript 5.3 - 编译为JavaScript
- esbuild - Vitest使用，用于快速转换

## 关键依赖

**关键：**
- commander 11.x - CLI参数解析和命令结构
- chalk 5.x - 终端输出样式
- fs-extra 11.x - 扩展的文件系统操作

**基础设施：**
- Node.js内置模块 - fs, path, child_process用于文件操作

## 配置

**环境：**
- 不需要环境变量
- 仅通过CLI标志配置

**构建：**
- `tsconfig.json` - TypeScript编译器选项
- `vitest.config.ts` - 测试运行器配置

## 平台要求

**开发：**
- macOS/Linux/Windows（任何具有Node.js的平台）
- 无外部依赖

**生产：**
- 作为npm包分发
- 通过npm install -g全局安装
- 在用户的Node.js安装上运行

---

*栈分析：2025-01-20*
*主要依赖更改后更新*
```
</good_examples>

<guidelines>
**STACK.md中应包含的内容：**
- 语言和版本
- 运行时要求（Node、Bun、Deno、浏览器）
- 包管理器和锁文件
- 框架选择
- 关键依赖（限制为5-10个最重要）
- 构建工具
- 平台/部署要求

**这里不应包含的内容：**
- 文件结构（那是STRUCTURE.md）
- 架构模式（那是ARCHITECTURE.md）
- package.json中的每个依赖（只有关键的）
- 实现细节（ defer到代码）

**填写此模板时：**
- 检查package.json中的依赖
- 从.nvmrc或package.json engines记录运行时版本
- 只包含影响理解的依赖（不是每个工具）
- 仅当版本重要时指定版本（破坏性更改、兼容性）

**阶段规划时有用：**
- 添加新依赖（检查兼容性）
- 升级框架（知道使用什么）
- 选择实现方法（必须与现有栈配合）
- 理解构建要求
</guidelines>
