# PROJECT.md 模板

用于 `.planning/PROJECT.md` — 活的项目上下文文档。

<template>

```markdown
# [项目名称]

## 这是什么的说明

[当前准确的描述 — 2-3 句话。这个产品做什么？为谁服务？
使用用户的语言和框架。每当现实与描述偏离时更新。]

## 核心价值

[最重要的那件事。如果其他一切都失败了，这个必须工作。
一个在出现权衡时驱动优先级的一句话。]

## 需求

### 已验证

<!-- 已发布并确认有价值。 -->

（尚未 — 发布以验证）

### 活跃

<!-- 当前范围。正在向这些目标构建。 -->

- [ ] [需求 1]
- [ ] [需求 2]
- [ ] [需求 3]

### 范围之外

<!-- 明确边界。包含防止重新添加的理由。 -->

- [排除项 1] — [为什么]
- [排除项 2] — [为什么]

## 上下文

[影响实现的背景信息：
- 技术环境或生态系统
- 相关的先前工作或经验
- 用户研究或反馈主题
- 需要解决的问题]

## 约束条件

- **[类型]**: [什么] — [为什么]
- **[类型]**: [什么] — [为什么]

常见类型: 技术栈、时间线、预算、依赖、兼容性、性能、安全

## 关键决策

<!-- 贯穿项目生命周期添加的约束未来工作的决策。 -->

| 决策 | 理由 | 结果 |
|----------|-----------|---------|
| [选择] | [为什么] | [✓ 好 / ⚠️ 重新审视 / — 待定] |

---
*上次更新: [日期] [触发原因后]*
```

</template>

<guidelines>

**这是什么的说明:**
- 产品的当前准确描述
- 2-3 句话捕获它的功能和目标用户
- 使用用户的词语和框架
- 当产品发展超出此描述时更新

**核心价值:**
- 单一最重要的事情
- 其他一切都可以失败；这个不能
- 在出现权衡时驱动优先级
- 很少变化；如果变化，那是重大转变

**需求 — 已验证:**
- 已发布并证明有价值的需求
- 格式: `- ✓ [需求] — [版本/阶段]`
- 这些是锁定的 - 更改它们需要明确讨论

**需求 — 活跃:**
- 当前正在构建的范围
- 直到发布和验证前，这些是假设
- 发布时移至已验证，如果被证伪则移至范围之外

**需求 — 范围之外:**
- 我们不构建的明确边界
- 总是包含理由（防止以后重新添加）
- 包括: 考虑过但拒绝的，推迟到未来，明确排除的

**上下文:**
- 影响实现决策的背景
- 技术环境、先前工作、用户反馈
- 需要解决的问题或技术债务
- 随着新上下文的出现而更新

**约束条件:**
- 实现选择的硬性限制
- 技术栈、时间线、预算、兼容性、依赖
- 包含"为什么" - 没有理由的约束会被质疑

**关键决策:**
- 影响未来工作的重要选择
- 在项目期间做出决策时添加
- 已知结果时跟踪：
  - ✓ 好 - 证明决策正确
  - ⚠️ 重新审视 - 可能需要重新考虑
  - — 待定 - 评估为时过早

**上次更新:**
- 总是注明何时以及为什么更新文档
- 格式: `在阶段 2 后` 或 `在 v1.0 里程碑后`
- 触发内容是否仍然准确的审查

</guidelines>

<evolution>

PROJECT.md 在项目生命周期中不断发展。

**每次阶段转换后:**
1. 需求被证伪？→ 移至范围之外并说明理由
2. 需求被验证？→ 移至已验证并引用阶段
3. 出现新需求？→ 添加到活跃需求中
4. 需要记录的决策？→ 添加到关键决策中
5. "这是什么的说明"仍然准确？→ 如果偏离则更新

**每次里程碑后:**
1. 所有部分全面审查
2. 核心价值检查 - 仍然正确的优先级？
3. 范围之外审查 - 理由仍然有效？
4. 用当前状态（用户、反馈、指标）更新上下文

</evolution>

<brownfield>

对于现有代码库：

1. **首先映射代码库** 通过 `/gsd:map-codebase`

2. **推断已验证需求** 从现有代码中：
   - 代码库实际做什么？
   - 建立了什么模式？
   - 什么明显有效并依赖？

3. **从用户收集活跃需求**：
   - 呈现推断的当前状态
   - 询问他们想构建什么

4. **初始化:**
   - 已验证 = 从现有代码推断
   - 活跃 = 用户对此工作的目标
   - 范围之外 = 用户指定的边界
   - 上下文 = 包括当前代码库状态

</brownfield>

STATE.md 引用 PROJECT.md:


这确保 Claude 读取当前的 PROJECT.md 上下文。

</state_reference>
