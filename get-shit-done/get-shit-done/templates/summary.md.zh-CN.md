# 总结模板

用于 `.planning/phases/XX-name/{phase}-{plan}-SUMMARY.md` — 阶段完成文档。

---

## 文件模板

```markdown
---

# 依赖关系图
requires:
  - phase: [依赖的前一个阶段]
    provides: [该阶段构建的可用的内容]
provides:
  - [此阶段构建/交付内容的清单]
affects: [需要此上下文的阶段名称或关键词列表]

# 技术跟踪
tech-stack:
  added: [本阶段添加的库/工具]
```

<frontmatter_guidance>
**目的：** 通过依赖关系图实现自动上下文组装。Frontmatter 使总结元数据机器可读，使 plan-phase 可以快速扫描所有摘要并根据依赖关系选择相关内容。

**快速扫描：** Frontmatter 是前 ~25 行，可以低成本地扫描所有摘要而无需阅读完整内容。

**依赖关系图：** `requires`/`provides`/`affects` 在阶段之间创建显式链接，实现上下文选择的传递闭包。

**子系统：** 主要分类（认证、支付、UI、API、数据库、基础设施、测试）用于检测相关阶段。

**标签：** 可搜索的技术关键词（库、框架、工具）用于技术栈感知。

**关键文件：** PLAN.md 中 @context 引用的重要文件。

**模式：** 未来阶段应该维护的约定。

**填充：** Frontmatter 在 execute-plan.md 中创建总结时填充。参见 `<step name="create_summary">` 了解字段指导。
</frontmatter_guidance>

<one_liner_rules>
一句话必须是实质性的：

**好的：**
- "使用 jose 库的 JWT 认证与刷新旋转"
- "带有 User、Session 和 Product 模型的 Prisma schema"
- "通过服务器发送事件的实时指标仪表板"

**坏的：**
- "阶段完成"
- "认证已实现"
- "基础架构完成"
- "所有任务完成"

一句话应该告诉某人实际交付了什么。
</one_liner_rules>

<guidelines>
**Frontmatter：** 强制 - 完成所有字段。为未来规划实现自动上下文组装。

**一句话：** 必须是实质性的。"使用 jose 库的 JWT 认证与刷新旋转" 而不是 "认证已实现"。

**决策记录部分：**
- 执行期间做出关键决策及其理由
- 提取到 STATE.md 累积上下文中
- 如果没有偏差，使用"无 — 按指定计划执行"

**创建后：** STATE.md 更新位置、决策、问题。
</guidelines>
