# 阶段上下文模板

用于 `.planning/phases/XX-name/{phase_num}-CONTEXT.md` - 捕获阶段的实现决策。

**目的：** 记录下游代理需要的决策。研究者使用这个来知道调查什么。规划者使用这个来知道哪些选择是锁定还是灵活的。

**核心原则：** 区段不是预先定义的。它们是从这个阶段实际讨论中产生的内容。CLI 阶段有 CLI 相关区段，UI 阶段有 UI 相关区段。

**下游消费者：**
- `gsd-phase-researcher` — 读取决策以聚焦研究（例如，"卡片布局" → 研究卡片组件模式）
- `gsd-planner` — 读取决策以创建具体任务（例如，"无限滚动" → 任务包含虚拟化）

## 示例

**示例 1：视觉功能（帖子流）**

```markdown
# 阶段 3: 帖子流 - 上下文

**收集时间：** 2025-01-20
**状态：** 准备规划

<domain>
## 阶段边界

在可滚动流中显示关注用户的帖子。用户可以查看帖子和参与计数。创建帖子和交互是分开的阶段。

</domain>

<decisions>
## 实现决策

### 布局样式
- 基于卡片的布局，不是时间线或列表
- 每个卡片显示：作者头像、姓名、时间戳、完整帖子内容、互动计数
- 卡片有微妙阴影、圆角 — 现代感

### 加载行为
- 无限滚动，不是分页
- 移动端下拉刷新
- 顶部新帖子指示器（"3 个新帖子"）而不是自动插入

### 空状态
- 友好的插图 +"关注人来查看帖子"
- 根据兴趣建议 3-5 个账户关注

### Claude 的决策权
- 加载骨架设计
- 精确的间距和排版
- 错误状态处理

</decisions>

<canonical_refs>
## 规范引用

### 流显示
- `docs/features/social-feed.md` — 流要求、帖子卡片字段、互动显示规则
- `docs/decisions/adr-012-infinite-scroll.md` — 滚动策略决策、虚拟化要求

### 空状态
- `docs/design/empty-states.md` — 空状态模式、插图指南

</canonical_refs>

<specifics>
## 具体想法

- "我喜欢 Twitter 如何显示新帖子指示器而不干扰滚动位置"
- 卡片应该感觉像 Linear 的问题卡片 — 简洁，不杂乱

</specifics>

<deferred>
## 延迟想法

- 对帖子评论 — 阶段 5
- 收藏帖子 — 添加到待办事项

</deferred>

---

*阶段：03-post-feed*
*上下文收集时间：2025-01-20*
```

**示例 2：CLI 工具（数据库备份）**

```markdown
# 阶段 2: 备份命令 - 上下文

**收集时间：** 2025-01-20
**状态：** 准备规划

<domain>
## 阶段边界

将数据库备份到本地文件或 S3 的 CLI 命令。支持完整和增量备份。恢复命令是单独的阶段。

</domain>

<decisions>
## 实现决策

### 输出格式
- JSON 用于编程用途，表格格式用于人类阅读
- 默认为表格，--json 标志用于 JSON
- 详细模式（-v）显示进度，默认静默

### 标志设计
- 常用选项的短标志：-o（输出）、-v（详细）、-f（强制）
- 清晰的长标志：--incremental、--compress、--encrypt
- 必需：数据库连接字符串（位置参数或 --db）

### 错误恢复
- 网络失败时重试 3 次，然后失败并显示清晰消息
- --no-retry 标志快速失败
- 失败时删除部分备份（无损坏文件）

### Claude 的决策权
- 进度条实现
- 压缩算法选择
- 临时文件处理

</decisions>

<canonical_refs>
## 规范引用

### 备份 CLI
- `docs/features/backup-restore.md` — 备份要求、支持的后端、加密规范
- `docs/decisions/adr-007-cli-conventions.md` — 标志命名、退出码、输出格式标准

</canonical_refs>

<specifics>
## 具体想法

- "我希望它感觉像 pg_dump — 数据库人员熟悉"
- 应该在 CI 管道中工作（退出码，无交互提示）

</specifics>

<deferred>
## 延迟想法

- 计划备份 — 单独阶段
- 备份轮换/保留 — 添加到待办事项

</deferred>

---

*阶段：02-backup-command*
*上下文收集时间：2025-01-20*
```

**示例 3：组织任务（照片库）**

```markdown
# 阶段 1: 照片组织 - 上下文

**收集时间：** 2025-01-20
**状态：** 准备规划

<domain>
## 阶段边界

将现有照片库组织到结构化文件夹中。处理重复项并应用一致的命名。标记和搜索是单独的阶段。

</domain>

<decisions>
## 实现决策

### 分组标准
- 主要按年分组，然后按月分组
- 通过时间聚类检测事件（2 小时内的照片 = 同一个事件）
- 事件文件夹按日期 + 命名（如果可用）

### 重复项处理
- 保留最高分辨率版本
- 将重复项移动到 _duplicates 文件夹（不删除）
- 记录所有重复项决定以供审查

### 命名约定
- 格式：YYYY-MM-DD_HH-MM-SS_originalname.ext
- 保留原始文件名作为后缀以便搜索
- 处理名称冲突，使用递增后缀

### Claude 的决策权
- 精确的聚类算法
- 如何处理没有 EXIF 数据的照片
- 文件夹表情符号使用

</decisions>

<canonical_refs>
## 规范引用

### 组织规则
- `docs/features/photo-organization.md` — 分组规则、重复项策略、命名规范
- `docs/decisions/adr-003-exif-handling.md` — EXIF 提取策略、缺失元数据的备用方案

</canonical_refs>

<specifics>
## 具体想法

- "我能够按大致拍摄时间找到照片"
- 不删除任何内容 — 最坏情况，移动到审查文件夹

</specifics>

<deferred>
## 延迟想法

- 人脸检测分组 — 未来阶段
- 云端同步 — 目前超出范围

</deferred>

---

*阶段：01-photo-organization*
*上下文收集时间：2025-01-20*
```

## 指导原则

**此模板为下游代理捕获决策。**

输出应回答："研究者需要调查什么？规划者哪些选择被锁定？"

**好的内容（具体决策）：**
- "基于卡片的布局，不是时间线"
- "网络失败时重试 3 次，然后失败"
- "按年分组，然后按月分组"
- "JSON 用于编程用途，表格用于人类"

**坏的内容（过于模糊）：**
- "应该感觉现代和简洁"
- "良好的用户体验"
- "快速和响应"
- "易于使用"

**创建后：**
- 文件位于阶段目录：`.planning/phases/XX-name/{phase_num}-CONTEXT.md`
- `gsd-phase-researcher` 使用决策聚焦调查并读取 canonical_refs 知道研究什么文档
- `gsd-planner` 使用决策 + 研究创建可执行任务并读取 canonical_refs 验证对齐
- 下游代理不应该需要再次询问用户已捕获的决策

**关键 — 规范引用：**
- `<canonical_refs>` 部分是必需的。每个 CONTEXT.md 必须有一个。
- 如果您的项目有外部规范、ADR 或设计文档，请按主题分组列出完整相对路径
- 如果 ROADMAP.md 每个阶段列出了 `Canonical refs:`，提取并扩展那些
- 分散在决策中的内联引用如"见 ADR-019"对下游代理无用 — 他们需要专用部分中的完整路径和部分引用
- 如果不存在外部规范，明确说明 — 不要默默地省略该部分
