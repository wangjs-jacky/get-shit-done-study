# 需求模板

用于 `.planning/REQUIREMENTS.md` 的模板 — 定义"完成"的可检查需求。

<template>

- [ ] **AUTH-02**: 用户注册后收到邮件验证
- [ ] **AUTH-03**: 用户可以通过邮件链接重置密码
- [ ] **AUTH-04**: 用户会话在浏览器刷新后保持

### [类别 2]

- [ ] **[CAT]-01**: [需求描述]
- [ ] **[CAT]-02**: [需求描述]

### [类别 3]

- [ ] **[CAT]-01**: [需求描述]
- [ ] **[CAT]-02**: [需求描述]

## v2 需求

推迟到未来版本。被追踪但不在当前路线图中。

### [类别]

- **[CAT]-01**: [需求描述]
- **[CAT]-02**: [需求描述]

<guidelines>

**需求格式:**
- ID: `[类别]-[编号]` (AUTH-01, CONTENT-02, SOCIAL-03)
- 描述: 以用户为中心，可测试，原子化
- 复选框: 仅用于 v1 需求 (v2 尚未可执行)

**类别:**
- 从研究 FEATURES.md 类别派生
- 保持与领域约定一致
- 常见类别: 身份验证、内容、社交、通知、审核、支付、管理

**v1 vs v2:**
- v1: 承诺范围，将包含在路线图阶段中
- v2: 已知但推迟，不在当前路线图中
- v2 → v1 需要路线图更新

**范围之外:**
- 明确排除及其理由
- 防止后续"为什么不包含 X"的问题
- 来自研究的反功能应在此处记录警告

**可追溯性:**
- 初始为空，在路线图创建期间填充
- 每个需求恰好映射到一个阶段
- 未映射的需求 = 路线图缺口

**状态值:**
- 待处理: 未开始
- 进行中: 阶段处于活跃状态
- 完成: 需求已验证
- 阻塞: 等待外部因素

</guidelines>

<evolution>

**每个阶段完成后:**
1. 将覆盖的需求标记为完成
2. 更新可追溯性状态
3. 记录任何范围发生变化的需求

**路线图更新后:**
1. 验证所有 v1 需求仍被映射
2. 如果范围扩大，添加新需求
3. 如果范围缩小，将需求移至 v2/范围之外

**需求完成标准:**
- 需求是"完成"当:
  - 功能已实现
  - 功能已验证（测试通过，手动检查完成）
  - 功能已提交

</evolution>

# 需求: CommunityApp

**定义时间:** 2025-01-14
**核心价值:** 用户可以与兴趣相投的人分享和讨论内容

## v1 需求

### 身份验证

- [ ] **AUTH-01**: 用户可以使用邮箱和密码注册
- [ ] **AUTH-02**: 用户注册后收到邮件验证
- [ ] **AUTH-03**: 用户可以通过邮件链接重置密码
- [ ] **AUTH-04**: 用户会话在浏览器刷新后保持

### 个人资料

- [ ] **PROF-01**: 用户可以创建显示名称的个人资料
- [ ] **PROF-02**: 用户可以上传头像图片
- [ ] **PROF-03**: 用户可以撰写简介（最多500字符）
- [ ] **PROF-04**: 用户可以查看其他用户的个人资料

### 内容

- [ ] **CONT-01**: 用户可以创建文本帖子
- [ ] **CONT-02**: 用户可以上传带图片的帖子
- [ ] **CONT-03**: 用户可以编辑自己的帖子
- [ ] **CONT-04**: 用户可以删除自己的帖子
- [ ] **CONT-05**: 用户可以查看帖子信息流

### 社交

- [ ] **SOCL-01**: 用户可以关注其他用户
- [ ] **SOCL-02**: 用户可以取消关注用户
- [ ] **SOCL-03**: 用户可以点赞帖子
- [ ] **SOCL-04**: 用户可以评论帖子
- [ ] **SOCL-05**: 用户可以查看活动流（所关注用户的帖子）

## v2 需求

### 通知

- **NOTF-01**: 用户接收应用内通知
- **NOTF-02**: 用户为新关注者接收邮件
- **NOTF-03**: 用户为对自己帖子的评论接收邮件
- **NOTF-04**: 用户可以配置通知偏好

### 审核

- **MODR-01**: 用户可以举报内容
- **MODR-02**: 用户可以屏蔽其他用户
- **MODR-03**: 管理员可以查看被举报内容
- **MODR-04**: 管理员可以删除内容
- **MODR-05**: 管理员可以封禁用户

## 范围之外

| 功能 | 原因 |
|---------|--------|
| 实时聊天 | 高复杂度，不是核心社区价值 |
| 视频帖子 | 存储/带宽成本，推迟到 v2+ |
| OAuth 登录 | 邮箱/密码对 v1 足够 |
| 移动应用 | 先做网络版，移动版稍后 |

## 可追溯性

| 需求 | 阶段 | 状态 |
|-------------|-------|--------|
| AUTH-01 | 阶段 1 | 待处理 |
| AUTH-02 | 阶段 1 | 待处理 |
| AUTH-03 | 阶段 1 | 待处理 |
| AUTH-04 | 阶段 1 | 待处理 |
| PROF-01 | 阶段 2 | 待处理 |
| PROF-02 | 阶段 2 | 待处理 |
| PROF-03 | 阶段 2 | 待处理 |
| PROF-04 | 阶段 2 | 待处理 |
| CONT-01 | 阶段 3 | 待处理 |
| CONT-02 | 阶段 3 | 待处理 |
| CONT-03 | 阶段 3 | 待处理 |
| CONT-04 | 阶段 3 | 待处理 |
| CONT-05 | 阶段 3 | 待处理 |
| SOCL-01 | 阶段 4 | 待处理 |
| SOCL-02 | 阶段 4 | 待处理 |
| SOCL-03 | 阶段 4 | 待处理 |
| SOCL-04 | 阶段 4 | 待处理 |
