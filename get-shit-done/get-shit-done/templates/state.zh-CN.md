# 状态模板

用于 `.planning/STATE.md` — 项目的活体记忆。

---

## 文件模板

```markdown
# 项目状态

## 项目参考

参见: .planning/PROJECT.md（更新于 [日期]）

**核心价值:** [来自 PROJECT.md 核心价值的一句话]
**当前重点:** [当前阶段名称]

## 当前位置

阶段: [X] 中的 [Y]（[阶段名称]）
计划: 当前阶段中的 [A] 中的 [B]
状态: [准备规划 / 规划中 / 准备执行 / 进行中 / 阶段完成]
上次活动: [YYYY-MM-DD] — [发生了什么]

进度: [░░░░░░░░░░] 0%

## 性能指标

**速度:**
- 完成的计划总数: [N]
- 平均持续时间: [X] 分钟
- 总执行时间: [X.X] 小时

**按阶段:**

| 阶段 | 计划 | 总计 | 平均/计划 |
|-------|-------|-------|----------|
| - | - | - | - |

**最近趋势:**
- 最后 5 个计划: [持续时间]
- 趋势: [改善 / 稳定 / 恶化]

*每个计划完成后更新*

## 累积上下文

### 决策

决策记录在 PROJECT.md 关键决策表中。影响当前工作的近期决策：

- [阶段 X]: [决策摘要]
- [阶段 Y]: [决策摘要]

### 待办事项

[来自 .planning/todos/pending/ — 会话中捕获的想法]

尚无。

### 阻塞/关注点

[影响未来工作的问题]

尚无。

## 会话连续性

上次会话: [YYYY-MM-DD HH:MM]
停止于: [上次完成操作的描述]
恢复文件: [路径到 .continue-here*.md（如果存在），否则为"无"]
```

<purpose>

STATE.md 是跨越所有阶段和会话的项目短期记忆。

**它解决的问题:** 信息被捕获在总结、问题和决策中，但没有系统性消费。会话开始时没有上下文。

**解决方案:** 一个小文件，它：
- 在每个工作流程中首先读取
- 每次重要操作后更新
- 包含累积上下文的摘要
- 支持即时会话恢复

</purpose>

<lifecycle>

**创建:** 在 ROADMAP.md 创建后（初始化期间）
- 引用 PROJECT.md（读取当前上下文）
- 初始化空的累积上下文部分
- 将位置设置为"阶段 1 准备规划"

**读取:** 每个工作流程的第一步
- progress: 向用户呈现当前状态
- plan: 指导规划决策
- execute: 了解当前位置
- transition: 了解什么已完成

**写入:** 每次重要操作后
- execute: SUMMARY.md 创建后
  - 更新位置（阶段、计划、状态）
  - 记录新决策（详情在 PROJECT.md 中）
  - 添加阻塞/关注点
- transition: 阶段标记完成后
  - 更新进度条
  - 清理已解决的阻塞
  - 刷新项目参考日期

</lifecycle>

<sections>

### 项目参考
指向 PROJECT.md 获取完整上下文。包括：
- 核心价值（那件重要的事）
- 当前重点（哪个阶段）
- 上次更新日期（如果过时则触发重新读取）

Claude 直接读取 PROJECT.md 获取需求、约束和决策。

### 当前位置
我们现在的位置：
- 阶段 X 中的 Y — 哪个阶段
- 计划 A 中的 B — 阶段中的哪个计划
- 状态 — 当前状态
- 上次活动 — 最近发生的事情
- 进度条 — 整体完成的视觉指示器

进度计算:（完成的计划）/（所有阶段中的计划总数）× 100%

### 性能指标
跟踪速度以了解执行模式：
- 完成的计划总数
- 每个计划的平均持续时间
- 按阶段分解
- 最近趋势（改善/稳定/恶化）

每个计划完成后更新。

### 累积上下文

**决策:** 引用 PROJECT.md 关键决策表，加上近期决策摘要以快速访问。完整决策日志存在于 PROJECT.md 中。

**待办事项:** 通过 /gsd:add-todo 捕获的想法
- 待办事项计数
- 引用 .planning/todos/pending/
- 如果数量少则简要列出，多则计数（例如，"5 个待办事项 — 参见 /gsd:check-todos"）

**阻塞/关注点:** 来自"下一阶段就绪"部分
- 影响未来工作的问题
- 前缀添加来源阶段
- 解决后清除

### 会话连续性
支持即时恢复：
- 上次会话是什么时候
- 上次完成什么
- 是否有 .continue-here 文件可以从中恢复

</sections>

<size_constraint>

将 STATE.md 保持低于 100 行。

这是一个摘要，而不是存档。如果累积上下文变得太大：
- 在摘要中只保留 3-5 个最近的决策（完整日志在 PROJECT.md 中）
- 只保留活跃阻塞，移除已解决的

目标是"读一次，就知道我们在哪里" — 如果太长，就失败了。

</size_constraint>
