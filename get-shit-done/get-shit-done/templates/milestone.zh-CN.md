# 里程碑条目模板

完成里程碑时将此条目添加到 `.planning/MILESTONES.md`：

```markdown
## v[X.Y] [名称] (发布: YYYY-MM-DD)

**交付：** [一句话描述发布的内容]

**完成的阶段：** [X-Y]（总共 [Z] 个计划）

**主要成就：**
- [主要成就 1]
- [主要成就 2]
- [主要成就 3]
- [主要成就 4]

**统计：**
- [X] 文件创建/修改
- [Y] 代码行数（主要语言）
- [Z] 阶段，[N] 计划，[M] 任务
- [D] 从开始到发布的天数（或里程碑到里程碑）

**Git 范围：** `feat(XX-XX)` → `feat(YY-YY)`

**下一步：** [下一个里程碑目标的简要描述，或"项目完成"]

---
```

<structure>
如果 MILESTONES.md 不存在，创建包含标题的文件：

```markdown
# 项目里程碑：[项目名称]

[按时间倒序排列的条目 - 最新在前面]
```
</structure>

<guidelines>
**何时创建里程碑：**
- 初始 v1.0 MVP 发布
- 主要版本发布（v2.0、v3.0）
- 重要功能里程碑（v1.1、v1.2）
- 存档规划前（捕获发布的内容）

**不要为以下内容创建里程碑：**
- 单个阶段完成（正常工作流）
- 进行中的工作（等待发布）
- 不构成发布的小错误修复

**包括的统计：**
- 计算修改的文件数：`git diff --stat feat(XX-XX)..feat(YY-YY) | tail -1`
- 计算 LOC：`find . -name "*.swift" -o -name "*.ts" | xargs wc -l`（或相关扩展）
- 来自 ROADMAP 的阶段/计划/任务计数
- 从第一个阶段提交到最后一个阶段提交的时间线

**Git 范围格式：**
- 第一个里程碑提交 → 最后一个里程碑提交
- 示例：`feat(01-01)` → `feat(04-01)` 表示阶段 1-4
</guidelines>

<example>
```markdown
# 项目里程碑：WeatherBar

## v1.1 安全与优化 (发布: 2025-12-10)

**交付：** 通过 Keychain 集成和全面错误处理进行安全加固

**完成的阶段：** 5-6（总共 3 个计划）

**主要成就：**
- 将 API 密钥存储从明文迁移到 macOS Keychain
- 为网络故障实现全面的错误处理
- 添加了 Sentry 崩溃报告集成
- 修复了自动刷新定时器中的内存泄漏

**统计：**
- 23 个文件被修改
- 新增了 650 行 Swift 代码
- 2 个阶段，3 个计划，12 个任务
- 从 v1.0 到 v1.1 用了 8 天

**Git 范围：** `feat(05-01)` → `feat(06-02)`

**下一步：** v2.0 带 Widget 支持的 SwiftUI 重新设计

---

## v1.0 MVP (发布: 2025-11-25)

**交付：** 带有当前条件和 3 天预报的菜单栏天气应用

**完成的阶段：** 1-4（总共 7 个计划）

**主要成就：**
- 带有弹出 UI 的菜单栏应用（AppKit）
- 带自动刷新的 OpenWeather API 集成
- 带条件图标的当前天气显示
- 带最高/低温的 3 天预报列表
- 为分发进行了代码签名和公证

**统计：**
- 创建了 47 个文件
- 2,450 行 Swift 代码
- 4 个阶段，7 个计划，28 个任务
- 从开始到发布用了 12 天

**Git 范围：** `feat(01-01)` → `feat(04-01)`

**下一步：** v1.1 的安全审计和加固
```
</example>
