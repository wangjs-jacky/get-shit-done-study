# 继续格式

完成命令或工作流后展示下一步的标准格式。

## 核心结构

```
---

## ▶ 下一步

**{标识符}: {名称}** — {一行描述}

`{可复制的命令}`

<sub>`/clear` 先 → 刷新上下文窗口</sub>

---

**也可选择：**
- `{备选选项1}` — 描述
- `{备选选项2}` — 描述

---
```

## 格式规则

1. **总是显示是什么** — 名称 + 描述，不要只是命令路径
2. **从源代码获取上下文** - ROADMAP.md 用于阶段，PLAN.md `<objective>` 用于计划
3. **内联代码中的命令** — 反引号，易于复制粘贴，渲染为可点击链接
4. **`/clear` 解释** — 总是包含，保持简洁但解释原因
5. **"也可选择" 而不是 "其他选项"** - 听起来更像应用
6. **视觉分隔符** — `---` 上下来突出显示

## 变体

### 执行下一个计划

```
---

## ▶ 下一步

**02-03: 刷新令牌轮换** — 添加带有滑动过期时间的 /api/auth/refresh

`/gsd:execute-phase 2`

<sub>`/clear` 先 → 刷新上下文窗口</sub>

---

**也可选择：**
- 执行前审查计划
- `/gsd:list-phase-assumptions 2` — 检查假设

---
```

### 执行阶段中的最终计划

添加这是最后一个计划以及后续内容的说明：

```
---

## ▶ 下一步

**02-03: 刷新令牌轮换** — 添加带有滑动过期时间的 /api/auth/refresh
<sub>第 2 阶段中的最终计划</sub>

`/gsd:execute-phase 2`

<sub>`/clear` 先 → 刷新上下文窗口</sub>

---

**完成后：**
- 第 2 阶段 → 第 3 阶段转换
- 下一步：**第 3 阶段：核心功能** — 用户仪表板和设置

---
```

### 计划一个阶段

```
---

## ▶ 下一步

**第 2 阶段：认证** — JWT 登录流程和刷新令牌

`/gsd:plan-phase 2`

<sub>`/clear` 先 → 刷新上下文窗口</sub>

---

**也可选择：**
- `/gsd:discuss-phase 2` — 先收集上下文
- `/gsd:research-phase 2` — 调研未知内容
- 审查路线图

---
```

### 阶段完成，准备下一个

显示完成状态后再执行下一步：

```
---

## ✓ 第 2 阶段完成

3/3 计划已执行

## ▶ 下一步

**第 3 阶段：核心功能** — 用户仪表板、设置和数据导出

`/gsd:plan-phase 3`

<sub>`/clear` 先 → 刷新上下文窗口</sub>

---

**也可选择：**
- `/gsd:discuss-phase 3` — 先收集上下文
- `/gsd:research-phase 3` — 调研未知内容
- 查看第 2 阶段构建的内容

---
```

### 多个等价选项

当没有明确的主要行动时：

```
---

## ▶ 下一步

**第 3 阶段：核心功能** — 用户仪表板、设置和数据导出

**直接计划：** `/gsd:plan-phase 3`

**先讨论上下文：** `/gsd:discuss-phase 3`

**先调研未知内容：** `/gsd:research-phase 3`

<sub>`/clear` 先 → 刷新上下文窗口</sub>

---
```

### 里程碑完成

```
---

## 🎉 版本 v1.0 完成

所有 4 个阶段都已发布

## ▶ 下一步

**开始 v1.1** — 质询 → 研究 → 需求 → 路线图

`/gsd:new-milestone`

<sub>`/clear` 先 → 刷新上下文窗口</sub>

---
```

## 获取上下文

### 用于阶段（来自 ROADMAP.md）：

```markdown
### 第 2 阶段：认证
**目标**：JWT 登录流程和刷新令牌
```

提取：`**第 2 阶段：认证** — JWT 登录流程和刷新令牌`

### 用于计划（来自 ROADMAP.md）：

```markdown
计划：
- [ ] 02-03: 添加刷新令牌轮换
```

或来自 PLAN.md `<objective>`：

```xml
<objective>
添加带有滑动过期窗口的刷新令牌轮换。

目的：延长会话生命周期而不妥协安全性。
</objective>
```

提取：`**02-03: 刷新令牌轮换** — 添加带有滑动过期时间的 /api/auth/refresh`

## 反模式

### 不要：仅命令（无上下文）

```
## 继续

运行 `/clear`，然后粘贴：
/gsd:execute-phase 2
```

用户不知道 02-03 是关于什么的。

### 不要：缺少 /clear 解释

```
`/gsd:plan-phase 3`

运行 /clear 先。
```

没有解释原因。用户可能会跳过。

### 不要："其他选项" 语言

```
其他选项：
- 审查路线图
```

听起来像是事后想法。使用"也可选择："。

### 不要：用代码块来围住命令

```
```
/gsd:plan-phase 3
```
```

模板内部的代码块会产生嵌套歧义。使用内联反引号。
