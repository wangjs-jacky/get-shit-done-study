✨ Architecture

Command 与 Workflow 分离架构

GSD 框架采用两层架构：Command 负责声明，Workflow 负责实现。一种优雅的设计模式 💫

问题

框架如何让命令既易于理解，又易于维护？

核心思想

将"做什么"（Command）与"怎么做"（Workflow）分离 ✨
🌸 Command
薄薄的入口层，定义元数据和依赖关系
💜 Workflow
厚厚的实现层，包含具体步骤和业务逻辑

两层职责

层级文件位置职责内容
Commandcommands/gsd/*.md声明"做什么"name, description, allowed-tools
Workflowworkflows/*.md定义"怎么做"steps, conditions, bash 命令

Command 文件结构

📝 YAML
---
name: gsd:update
description: Update GSD to latest version
allowed-tools:
  - Bash
  - AskUserQuestion
---

设计亮点

🎯 关注点分离
接口与实现解耦
🔒 工具权限控制
每个命令明确声明需要的工具
♻️ 可复用
一个 workflow 可被多个 command 引用
🛠️ 可维护
修改实现不影响命令接口

可复用场景

  • jacky-skills：可以将 skill 定义与实现分离
  • 任何需要声明式接口 + 命令式实现的场景
